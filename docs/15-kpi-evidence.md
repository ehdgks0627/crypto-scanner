# KPI Evidence

이 문서는 발표용 KPI 중 실측값이 아닌 비교 기준선을 과장 없이 설명하기 위한 근거를 정리한다.

## 15.1 Before: 수동 grep 기준

`Before · 수동 grep 기준 — 수일 ~ 수주`는 자동화 전 환경에서 사람이 직접 암호 자산 목록을 만드는 경우의 정성 비교 기준이다. 이 값은 실제 벤치마크 시간이 아니라, 현재 데모 시드 범위를 사람이 수동으로 점검할 때 필요한 작업량을 보수적으로 산정한 것이다.

구조화된 산정표는 `docs/kpi/manual-grep-baseline.json`에 둔다. 현재 기준 범위는 다음과 같다.

| 항목 | 값 |
| --- | ---: |
| 암호 자산 | 67개 |
| 발견 엔드포인트 | 32개 |
| 내부 점검 후보 호스트 | 10개 |
| 스캔 대상 | 31개 |

수동 작업에는 대상 목록 확인, 접속 권한 준비, `grep`/`find` 기반 파일 탐색, `openssl x509`/`ssh-keygen` 기반 인증서와 키 파싱, STARTTLS·SSH·TLS별 프로토콜 확인, 중복 제거, CBOM 형태 기록, 검토와 인수인계가 포함된다.

산정 결과는 최소 34.45시간, 최대 72.25시간이다. 하루 8시간 기준 약 4.31~9.03 인일이며, 실제 조직에서는 접속 권한, 검토 대기, 운영 담당자 확인 시간이 끼기 때문에 발표에서는 `수일 ~ 수주`라는 정성 표현을 사용한다.

발표 문구는 다음 수준으로 제한한다.

> 같은 범위를 사람이 수동 grep과 명령행 도구로 확인하면 작업 자체만 약 4~9 인일이 필요하고, 접근 권한과 검토 대기까지 포함하면 수일에서 수주까지 늘어날 수 있습니다.

주의할 점은 이 값이 자동화 도구의 실측 성능과 직접 비교되는 벤치마크가 아니라는 것이다. 자동화 이후의 `수 분` 또는 `10분 이내` 주장은 별도 실행 시간 측정값으로만 말해야 한다.

## 15.2 After: 시스템 실행 시간

`After · 우리 시스템 — 수 분`은 최신 CBOM 스냅샷과 연결된 Scan Job의 `started_at`과 `finished_at` 차이로 계산한다. 대시보드의 `automated_inventory_runtime_minutes_per_scan` KPI는 이 값을 분 단위로 표시한다. 구조화된 근거는 `docs/kpi/runtime-minutes-evidence.json`에 둔다.

현재 데모 시드는 25개 서비스 테스트베드의 최신 스캔 작업을 6분 실행으로 기록한다. 이 값은 수동 기준선과 달리 DB에 저장된 작업 timestamp에서 직접 계산되므로 발표에서는 `시스템에서는 같은 데모 범위의 자동 스캔 결과가 6분 단위로 확인된다`고 표현할 수 있다.

단, 이 값은 데모 시드 기준의 실행 시간이며 실제 운영 환경에서는 네트워크 지연, 타임아웃, Agent 응답성, 대상 수, 선택한 scanner 수에 따라 달라질 수 있다.

## 15.3 전체 파이프라인 실행 시간

`전체 파이프라인 — 10분 이내`는 데모 시드에서 대표 단계로 잡은 Discovery, Scan, Recompute 작업의 `started_at`과 `finished_at` 차이를 합산해 계산한다. 대시보드의 `full_pipeline_runtime_minutes` KPI는 최신 CBOM 스냅샷과 연결된 Scan Job의 `scenario`를 기준으로 같은 시나리오의 완료된 Discovery/Recompute 작업을 찾아 합산한다.

현재 데모 시드는 Discovery 2분, Scan 6분, Recompute 1분으로 기록되어 전체 파이프라인 실행 시간이 9분으로 표시된다. 이 값은 시연 가능한 시간 안에 탐색, 자산화, 위험 재계산까지 한 흐름으로 확인할 수 있음을 보여주기 위한 발표용 기준이다.

1. 실제 사이트 시연 흐름
===============================================================================

[~] 1.1 대시보드에서 발표 시작이 가능해야 한다.
        현재:
        - 대시보드 진입과 시연 데이터 로드 버튼은 있다.
        - 대시보드 진입 시 브라우저에 저장된 과거 스냅샷 선택값이 있어도
          최신 스냅샷으로 전역 선택값을 갱신하고 최신 KPI를 조회한다.
        보완 필요:
        - 발표자료 수치와 대시보드 KPI가 일치하는지 검증

[~] 1.2 탐색 대상 화면에서 "찾기" 단계를 설명할 수 있어야 한다.
        현재:
        - CIDR, 특정 IP / 도메인 입력 UI가 있다.
        - 지원 서비스 체크박스는 접힌 상태로 표시된다.
        - Discovery Agent 또는 중앙 Worker 실행 위치를 선택할 수 있다.
        - 같은 CIDR/IP/domain은 새 이력 row를 계속 쌓지 않고 등록된 탐색 대상의
          최신 실행 상태로 갱신한다.
        - 시연 데이터 로드 시 `172.20.0.0/16` 탐색 대상과 발견 엔드포인트가
          함께 생성된다.
        발표 기준:
        - 실제 Agent 실행이 필수는 아니다.
        보완 필요:
        - 발표 흐름에서는 "Agent를 실제 실행했다"가 아니라 "이 단계에서 탐색 대상을 관리한다"로 설명

[x] 1.3 발견 엔드포인트를 스캔 대상으로 자동 등록하는 흐름이 보여야 한다.
        현재:
        - 탐색이 완료되면 발견 엔드포인트가 별도 수동 승인 없이 스캔 대상으로 자동 등록된다.
        - 탐색 상세 화면은 엔드포인트별 스캔 대상 등록 상태와 Target ID를 보여준다.
        - 시연 데이터 기준 기본 탐색 대상 `172.20.0.0/16`에서 발견 엔드포인트 33개,
          자동 등록 스캔 대상 31개를 확인할 수 있다.

[~] 1.4 스캔 대상과 스캔 실행 화면이 자연스럽게 이어져야 한다.
        현재:
        - 스캔 대상 목록, 스캔 실행 생성, 스캐너 선택 UI가 있다.
        - 스캔 작업 상세의 결과 카드는 스냅샷과 식별 자산 수만 표시하고,
          탐색 작업/업데이트 점수 같은 다른 작업 유형 전용 필드는 숨긴다.
        보완 필요:
        - 가데이터 기준으로 스캔 대상 목록이 비어 있지 않아야 한다.
        - 발표 중 실제 스캔을 실행하지 않더라도 최근 스캔 실행 이력과 결과가 보여야 한다.

[~] 1.5 식별 자산 화면에서 CBOM 스냅샷을 설명할 수 있어야 한다.
        현재:
        - 식별 자산 목록, 자산 상세, CBOM JSON export, 스냅샷 비교 버튼이 있다.
        - 스냅샷이 모두 삭제된 상태에서는 브라우저에 남은 과거 스냅샷 선택값을
          상세 API 호출에 사용하지 않고 빈 상태 화면을 표시한다.
        보완 필요:
        - 시연 데이터의 자산 수, 알고리즘 분포, 자산 상세 예시가 발표자료와 맞아야 한다.
        - `srv-01` 같은 대표 자산 라벨이 실제 자산 목록/상세에 노출되어야 한다.

[~] 1.6 위험평가 화면에서 우선순위를 설명할 수 있어야 한다.
        현재:
        - 위험 점수, 등급, DHS 우선순위, 가중치 조정, 재계산 UI가 있다.
        보완 필요:
        - 발표자료의 예시 점수와 실제 화면 점수를 통일해야 한다.
        - 현재 구현 기준은 `srv-01 = 9.2 / P1` 쪽이 안전하다.

[~] 1.7 Review Targets 화면에서 전환 후보를 설명할 수 있어야 한다.
        현재:
        - 전환 후보 목록, 목표 알고리즘, 영향도, 보고서 다운로드가 있다.
        - 화면에 "실제 인증서 재발급/키 교체/설정 변경은 수행하지 않는다"는 안내가 있다.
        보완 필요:
        - 발표자료 표현을 `교체`가 아니라 `전환 후보 선정 및 계획 추천`으로 맞춘다.
        - 목표 알고리즘 추천 수치와 대표 예시가 발표자료와 맞는지 확인

[~] 1.8 가용성 검사 화면에서 "확인하기" 단계를 설명할 수 있어야 한다.
        현재:
        - 가용성 검사 실행 목록, 자산별 결과, 성공률, 지연, 실패율 UI가 있다.
        보완 필요:
        - 가데이터 기준으로 가용성 검사 실행이 실제 화면에 표시되어야 한다.
        - 발표자료의 `handshake 100%`, `latency +12ms`, `failure 0` 같은 수치가
          실제 가용성 검사 화면의 데이터와 맞는지 확인
        - 실제 PQC 전환 후 검증으로 말하지 않고, 전환 계획 적용 시 확인해야 할
          가용성 지표로 설명한다.

[x] 1.9 설정 화면에서 시연 데이터를 정리할 수 있어야 한다.
        구현:
        - 설정 화면에 스냅샷/식별 자산 삭제 버튼을 추가했다.
        - 설정 화면에 스캔 대상 삭제 버튼을 추가했다.
        - 스냅샷/식별 자산 삭제 시 CBOM 스냅샷, 식별 자산, 위험 점수,
          가용성 검사, 스캔 작업 이력을 함께 정리한다.
        - 스캔 대상 삭제 시 Agent 등록 정보는 유지한다.

[x] 1.10 홈페이지 기반 컨텍스트 추론을 대시보드에서 확인할 수 있어야 한다.
        구현:
        - 웹 서비스 스캔 이후 공개 홈페이지의 title, meta description, 본문 키워드로
          서비스 역할, 민감도, 중요도, 노출 범위, 보호 기간을 제한적으로 추정한다.
        - 원문 HTML이나 민감한 파일을 저장하지 않고, 추론 근거로 title/description,
          매칭된 키워드, 신뢰도만 Target context에 저장한다.
        - 대시보드에 `홈페이지 추론 컨텍스트` 카드를 추가해 대상별 추론 결과를 확인한다.
        - CBOM export에 `context.*`, `context.homepage.*` 속성을 포함한다.
        - 시연 데이터 로드 시 대표 웹 서비스의 홈페이지 추론 결과가 함께 표시된다.

[x] 1.11 자산 상세 화면에서 Enriched CBOM을 확인할 수 있어야 한다.
        구현:
        - 자산 상세 API에 단일 자산 기준 `enriched_cbom_component`를 추가했다.
        - CBOM export와 같은 컴포넌트 생성 로직을 재사용해 cryptoProperties,

        - `/snapshots/{snapshotId}/assets/{assetId}` 화면 최하단에 Enriched CBOM
          단일 table을 표시한다.
        - BOM ref, component type, name, cryptoProperties, properties를 같은 table
          안에 합쳐 중복 요약/JSON 원문 표시는 제거했다.

[x] 1.12 자산 컨텍스트 수정 시 기존 적용값을 함께 확인할 수 있어야 한다.

        - 컨텍스트 재정의 입력 항목에 현재 적용값과 출처를 함께 표시한다.
        - 값이 비어 있는 입력은 `재정의 없음`으로 표현해 현재 적용값을 유지한다는

[ ] 2.1 실제 대시보드 시드 데이터를 발표자료 기준 수치로 맞춘다.
        맞출 수 있는 항목:

        - 식별 자산 47개
        - P1 12개

        - 실패 0건
        현재 위험:

        완료 기준:
        - `/api/dashboard/demo-seed` 실행 후 실제 사이트 화면의 수치가 발표자료와 일치한다.

[ ] 2.2 대표 자산 `srv-01` 수치를 하나로 통일한다.
        선택지:
        - 구현을 `9.6 / 10 · P1`로 맞춘다.
        - 발표자료를 `9.2 / 10 · P1`로 수정한다.
        권장:
        - 이미 구현/테스트에서 쓰던 `9.2 / P1` 기준으로 발표자료를 낮추는 편이 안전하다.

[ ] 2.3 Agent 기반 발견 수치를 실제 발표 표현과 맞춘다.
        현재 위험:
        - 발표자료가 `Discovery Agent 28개`, `Host Agent 24개`, `중복 5개`처럼
          실제 Agent 실행 결과처럼 읽힐 수 있다.
        발표 기준:
        - Agent를 실제 실행하지 않아도 된다는 조건이 있으므로,
          `시연 데이터 기준 발견 출처` 또는 `가데이터 기준 수집 출처`로 표현한다.
        구현 선택:
        - 실제 화면에서는 source/source_label 같은 필드를 통해 발견 출처를 보여준다.

[~] 2.4 알고리즘 추천 주체를 정확히 맞춘다.
        현재:
        - 실제 추천은 규칙 기반 매핑에 가깝다.
        - 자산별 정성 위험평가와 전환 권고 문장은 LLM provider를 통해 생성할 수 있다.
        - `codex-cli` provider를 추가해 `codex exec`로 LLM 호출이 가능하다.
        - Docker 이미지에는 Codex CLI가 포함되며, 실제 활성화는
          `docker-compose.codex.yml`과 host Codex 인증 정보 마운트를 사용한다.
        발표자료 수정 필요:
        - `LLM이 PQC 알고리즘을 자동 추천`이 아니라
          `위험 근거 설명은 보조하고, 전환 후보 알고리즘은 정책/규칙 기반으로 추천`으로 표현한다.

3. 발표자료에서 반드시 낮춰 말해야 하는 항목
===============================================================================

[ ] 3.1 `PQC 알고리즘으로 교체` 표현을 수정한다.
        이유:
        - 현재 시스템은 실제 키 교체, 인증서 재발급, 서버 설정 변경, 서비스 재시작을 수행하지 않는다.
        권장 표현:
        - `전환 후보 선정`
        - `PQC 매핑 계획 수립`
        - `우선 전환 검토 대상 제시`

[ ] 3.2 `전환 후 검증` 표현을 수정한다.
        이유:
        - 현재는 실제 전환 전/후 환경을 자동 구성해 비교하는 단계가 아니다.
        권장 표현:
        - `전환 계획 적용 시 확인해야 할 가용성 지표`
        - `서비스 연결 가능 여부 확인`
        - `가용성 검사 결과 조회`

[ ] 3.3 `스캔 한 번으로 실제 발견` 표현을 수정한다.
        이유:
        - 발표에서는 가데이터 주입을 허용하기로 했으므로 실시간 발견처럼 말하면 과장이다.
        권장 표현:
        - `시연 데이터 기준`
        - `재현 가능한 테스트베드 결과 기준`
        - `실제 사이트 화면에서 결과를 확인`
